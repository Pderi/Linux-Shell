## 一、项目总览（共同部分）

### 1.1 Shell 是什么

Shell 是用户与 Linux 内核之间的**命令解释器**。你在终端敲一行字，Shell 负责把这句话"翻译"成内核能理解的操作。整个过程可以概括为三步：

1. **读入**：从终端读取一行用户输入
2. **解析**：把字符串拆成"命令名 + 参数 + 特殊符号（管道、重定向等）"
3. **执行**：调用对应的系统功能完成操作

```
用户输入: ls -l | grep hello > out.txt
         │
         ▼
    ┌─────────────────────────────────────────┐
    │  myshell                                │
    │  ① input_read_line()    读入这一行       │
    │  ② parse_line()         拆成两段命令     │
    │  ③ execute_pipeline()   fork+pipe+dup2  │
    │     ├─ 子进程1: builtin_ls("-l")        │
    │     └─ 子进程2: builtin_grep("hello")   │
    │        └─ 输出重定向到 out.txt          │
    └─────────────────────────────────────────┘
         │
         ▼
    Linux Kernel（文件系统、进程管理等）
```

### 1.2 核心设计原则：系统调用可以，系统命令不行

这是本项目最重要的设计约束：

| ✅ 允许 | ❌ 禁止 |
|---------|---------|
| **系统调用**：`open`、`read`、`write`、`stat`、`opendir`、`getcwd`、`chdir`、`fork`、`pipe`、`dup2` 等 | **系统命令**：通过 `execvp` 调用 `/bin/ls`、`/bin/cat`、`/bin/grep` 等外部程序 |
| **C 标准库函数**：`fopen`、`printf`、`getline`、`strstr` 等 | `system()`、`popen()` 等套壳函数 |

**为什么？** 系统调用是内核直接提供的服务接口，是"原生函数"；而 `execvp("/bin/ls")` 只是套壳调用了系统自带的 `ls` 程序，没有体现对操作系统原理的理解。本项目所有命令（`ls`/`cat`/`grep`/`pwd` 等）都在 Shell 进程内用系统调用**从头实现**。

### 1.3 三人协作的整体流程

一条命令从输入到执行，依次经过三人的模块：

```
┌──────────────────────────────────────────────────────────────┐
│  秦健超：input_read_line()     ← 用 readline 读入一行        │
│  郭沛特：alias_expand_line()    ← 展开别名（如 ll → ls -l）    │
│  秦健超：parse_line()          ← 解析为 Pipeline 结构         │
│  张孙毓：history_add()         ← 记录原始输入到历史             │
│  秦健超：shell_execute()       ← 分发：单命令 or 管道          │
│      └─ 郭沛特：run_builtin()  ← 执行内建命令（系统调用实现）   │
│      └─ 张孙毓：execute_pipeline() ← 管道/后台 fork 执行       │
└──────────────────────────────────────────────────────────────┘
```

### 1.4 公共数据结构

整个项目围绕两个核心结构体运作，定义在 `include/parser.h`：

```c
// include/parser.h

#define MAX_PIPELINE 16   // 管道最多 16 段
#define MAX_ARGS     64   // 每条命令最多 64 个参数

typedef struct {
    char *argv[MAX_ARGS];   // 参数数组，argv[0] 是命令名，以 NULL 结尾
    int   argc;             // 参数个数
    char *in_file;          // 输入重定向文件：< file（NULL 表示无）
    char *out_file;         // 输出重定向文件：> file 或 >> file
    int   append;           // 1=追加模式(>>)，0=覆盖模式(>)
} Command;

typedef struct {
    Command cmds[MAX_PIPELINE]; // 管道各段命令
    int     count;              // 管道段数（至少 1）
    int     background;         // 1=行末有 &，后台运行
} Pipeline;
```

**用一个真实例子来理解：** 用户输入 `grep hello a.txt | grep hello > out.txt &`

解析后的 Pipeline 结构如下：

```
Pipeline {
    count = 2,          // 两段命令，用 | 分隔
    background = 1,     // 行末有 &

    cmds[0] = {          // 第一段：grep hello a.txt
        argv: ["grep", "hello", "a.txt", NULL],
        argc: 3,
        in_file: NULL,       // 没有 < 重定向
        out_file: NULL,      // 没有直接 > 重定向（输出进入管道）
        append: 0
    },

    cmds[1] = {          // 第二段：grep hello，输出重定向到 out.txt
        argv: ["grep", "hello", NULL],
        argc: 2,
        in_file: NULL,
        out_file: "out.txt",  // > out.txt
        append: 0            // 覆盖模式
    }
}
```

这个结构是三人协作的"交接物"：秦健超负责生成它，张孙毓负责执行它，郭沛特的命令被它调用。

### 1.5 涉及的 Linux 原理（前置知识）

在深入代码之前，先讲清楚本项目用到的几个核心 Linux 概念。理解这些原理，代码就不再晦涩。

#### ① 文件描述符（File Descriptor, fd）

**一句话：Linux 中"一切皆文件"，文件描述符就是操作这些"文件"的编号。**

每个进程有一张**文件描述符表**，记录它打开的所有文件。表是一个整数数组，0/1/2 是预留的：

| fd 编号 | 名称 | 指向 |
|---------|------|------|
| 0 | `STDIN_FILENO` | 标准输入（默认是终端键盘） |
| 1 | `STDOUT_FILENO` | 标准输出（默认是终端屏幕） |
| 2 | `STDERR_FILENO` | 标准错误（默认是终端屏幕） |
| 3+ | | `open()` 新打开的文件、管道、网络套接字等 |

**为什么 Shell 要管 fd？** 因为重定向和管道的本质就是"换掉 fd 指向的文件"：
- `echo hi > file`：把 fd 1 从"终端屏幕"换成"file 文件"，之后所有写到 stdout 的内容都进了文件
- `cmd1 | cmd2`：把 cmd1 的 fd 1 指向管道写端，cmd2 的 fd 0 指向管道读端

#### ② dup2() — "偷梁换柱"的系统调用

`dup2(oldfd, newfd)` 的作用是：**把 oldfd 指向的文件，复制到 newfd 的位置**。之后操作 newfd 就等于操作 oldfd 指向的文件。

```
调用前：                          调用 dup2(3, 1) 后：
fd 0 → 终端（stdin）              fd 0 → 终端（stdin）
fd 1 → 终端（stdout）             fd 1 → file.txt ← 被替换！
fd 2 → 终端（stderr）             fd 2 → 终端（stderr）
fd 3 → file.txt                   fd 3 → file.txt（仍然存在）
```

之后 `printf("hello")` 写到 fd 1（stdout），实际写入了 `file.txt`。这就是重定向的原理。

**恢复方法：** 在重定向前先用 `dup(1)` 备份原始 stdout，执行完后用 `dup2(saved, 1)` 恢复。本项目 `run_builtin_in_parent()` 就是这么做的。

> **为什么前台内建 + 重定向不 fork？** 若 fork 子进程跑 `echo > file`，子进程很快 `_exit`，`SIGCHLD` handler 可能抢先 `waitpid` 回收，父进程再 wait 就会报 `ECHILD: No child processes`。在父进程内临时 dup2、执行、恢复，既高效又避免竞态（见 1.5 ⑭）。

#### ③ fork() — 创建子进程

`fork()` 是 Unix 最核心的系统调用之一：**创建一个与当前进程几乎完全相同的子进程**。

```
fork() 调用前：          fork() 调用后：
                         ┌─ 父进程（PID=1000）
                         │   fork() 返回子进程 PID（1001）
只有一个进程              │
PID=1000                 └─ 子进程（PID=1001）
                             fork() 返回 0
                             内存是父进程的副本
```

**返回值的妙用：** `fork()` 在父进程中返回子进程的 PID（>0），在子进程中返回 0。通过判断返回值，可以让父子进程执行不同的代码：

```c
pid_t pid = fork();
if (pid == 0) {
    // 子进程执行这里
} else {
    // 父进程执行这里
}
```

**为什么 Shell 要 fork？** 
- 管道：每段命令需要一个独立进程，各自有自己的 fd 表，才能独立设置 dup2
- 后台：fork 后父进程不等子进程，立即返回提示符；子进程在后台独立运行

#### ④ waitpid() — 等待子进程结束

`waitpid(pid, &status, 0)` 让父进程**阻塞等待**指定子进程结束，并获取其退出状态。

| 参数 | 含义 |
|------|------|
| `pid` | 要等待的子进程 PID（-1 表示任意子进程） |
| `&status` | 接收子进程退出状态 |
| `0` | 阻塞等待；`WNOHANG` 表示非阻塞（立即返回） |

**僵尸进程（Zombie）：** 子进程退出后，内核保留它的退出状态，直到父进程调用 `waitpid` 回收。如果父进程不回收，子进程就变成"僵尸进程"——已死但占着进程表条目。后台命令的子进程就是靠 SIGCHLD 处理函数自动回收的。

#### ⑤ pipe() — 进程间通信

`pipe(fds[2])` 创建一条**匿名管道**，返回两个文件描述符：

| fd | 端 | 作用 |
|----|----|------|
| `fds[0]` | 读端 | 从管道读数据（读不到数据时阻塞，直到写端有数据或全部关闭） |
| `fds[1]` | 写端 | 向管道写数据（管道满时阻塞） |

**关键特性：** 当所有写端都关闭后，读端的 `read()` 返回 0（EOF）。这就是为什么管道场景中，父子进程都要关闭不需要的管道 fd——否则读端永远等不到 EOF。

```
cmd1 | cmd2 的管道连接：

cmd1 子进程                    cmd2 子进程
stdout(fd 1) ──→ pipe 写端    pipe 读端 ──→ stdin(fd 0)
                (fds[1])       (fds[0])
```

#### ⑥ 信号（Signal）— 软中断

信号是内核通知进程事件的一种机制。本项目用到的关键信号：

| 信号 | 触发时机 | 本项目用途 |
|------|----------|-----------|
| `SIGCHLD` | 子进程退出时内核发给父进程 | 处理函数中 `waitpid(-1, WNOHANG)` 回收后台子进程 |
| `SIGINT` | 用户按 Ctrl+C | 子进程恢复默认处理（终止） |
| `SIGTSTP` | 用户按 Ctrl+Z | 子进程恢复默认处理 |

**竞态问题：** 前台 `waitpid(pid, ...)` 和 SIGCHLD 处理函数可能竞争同一个子进程。如果处理函数先回收了子进程，`waitpid` 就会返回 `ECHILD`（No child processes）。本项目用 `sigprocmask` 在 wait 期间屏蔽 SIGCHLD 来解决。

#### ⑦ 目录操作：opendir / readdir / stat

这三个系统调用实现了 `ls` 命令的核心：

| 系统调用 | 作用 | 类比 |
|----------|------|------|
| `opendir(path)` | 打开一个目录，返回 `DIR*` 句柄 | 像打开一本书 |
| `readdir(dir)` | 逐个读取目录项，返回 `dirent*`（含文件名） | 像翻书页 |
| `stat(path, &st)` | 获取文件元数据（大小、权限、时间等） | 像查目录卡片 |

`struct stat` 包含：`st_mode`（文件类型+权限）、`st_size`（大小）、`st_mtime`（修改时间）、`st_nlink`（硬链接数）等。

#### ⑧ 环境变量

环境变量是进程的"全局配置"，用 `getenv("HOME")` 读取。每个进程继承父进程的环境变量。Shell 中 `cd` 无参数时用 `getenv("HOME")` 获取家目录路径。`echo $HOME` 也是通过 `getenv` 展开。

#### ⑨ open / read / write / close — 文件 I/O 四件套

| 系统调用 | 作用 | 本项目用途 |
|----------|------|-----------|
| `open(path, flags, mode)` | 打开/创建文件，返回 fd | 重定向、`cat` 读文件 |
| `read(fd, buf, n)` | 从 fd 读最多 n 字节 | `cat` 逐块读取 |
| `write(fd, buf, n)` | 向 fd 写 n 字节 | `cat` 输出到 stdout |
| `close(fd)` | 关闭 fd，释放内核资源 | dup2 后关闭多余 fd |

**与 C 标准库的关系：** `fopen`/`fread`/`fprintf` 底层仍走上述系统调用，只是加了缓冲。本项目 `cat` 用 `open+read+write` 直接操作 fd，更贴近内核；`grep` 用 `fopen+getline` 是为了简化按行读取。

**权限位 `0644`：** `open(..., O_CREAT, 0644)` 中 `0644` 表示 owner 可读写、group/other 只读（八进制权限）。

#### ⑩ _exit() vs exit() — 子进程必须用 _exit

| 函数 | 行为 | 使用场景 |
|------|------|----------|
| `exit(code)` | C 库函数：刷新 stdio 缓冲、调用 `atexit` 处理器、再 `_exit` | 普通 C 程序 main 返回 |
| `_exit(code)` | 系统调用：直接终止进程，不刷新缓冲 | **fork 出的子进程** |

**为什么子进程用 `_exit`？** fork 后子进程复制了父进程的 stdio 缓冲。若子进程 `exit()`，可能把缓冲里尚未输出的数据重复刷到终端或管道，造成输出错乱。本项目 `run_command_child()` 和管道子进程统一 `_exit(rc)`。

#### ⑪ nanosleep() — sleep 内建的实现

`sleep 2` 不是 busy-wait 循环，而是调用 `nanosleep()` 让出 CPU：

```c
struct timespec ts = { .tv_sec = 2, .tv_nsec = 0 };
nanosleep(&ts, &ts);   // 被信号打断时 ts 会被更新为剩余时间
```

若被 `SIGINT` 打断，返回 `EINTR`，循环重试剩余时间。这是可中断睡眠的标准写法。

#### ⑫ 内建命令 vs 外部命令 — 本项目的特殊设计

标准 bash 对外部命令走 `fork + execve`：子进程用新程序**替换**自身内存镜像。本项目**禁止 exec 外部程序**，所有命令在 Shell 进程内用系统调用实现：

```
标准 bash:  ls → fork → execve("/bin/ls") → 子进程变成 ls 程序
本项目:     ls → run_builtin() → opendir/readdir/stat（仍在 myshell 进程内）
```

因此 `type wc` 输出 `wc: not found`，而非 PATH 查找。管道也只允许内建命令串联。

#### ⑬ readline 与 Tab 补全原理

GNU readline 在交互模式下维护一行输入缓冲区 `rl_line_buffer`。按 Tab 时：

1. 调用 `rl_attempted_completion_function`（本项目注册为 `myshell_completion`）
2. 根据光标位置判断补全**命令名**还是**文件路径**
3. `rl_completion_matches(text, generator)` 收集所有前缀匹配的候选
4. **唯一匹配** → 一次 Tab 直接补全；**多个匹配** → 按两次 Tab 列出列表

`rl_basic_word_break_characters` 定义"词"的边界。若把 `'` 设为断词符，输入 `alias dir='ls -l'` 时 readline 会在引号处错误切分，导致补全失效——因此本项目将其移出断词符，改由 `rl_completer_quote_characters` 处理引号。

#### ⑭ SIGCHLD 竞态与 sigprocmask — 重定向 bug 的根因

```
时间线（有 bug 的旧方案）：
  父进程 fork 子进程跑 echo > file
  子进程很快 exit
  SIGCHLD → handler 用 waitpid(-1, WNOHANG) 回收子进程
  父进程再 waitpid(pid, ...) → ECHILD（子进程已被 handler 收走）
  perror 打印 "waitpid: No child processes"
```

**修复策略（两层）：**
1. **主方案：** 前台内建 + 重定向不 fork，在父进程 `dup2` → 执行 → 恢复（`run_builtin_in_parent`）
2. **兜底：** `wait_pid_blocking` 遇到 `ECHILD` 视为子进程已正常回收
3. **同步：** 阻塞 wait 期间用 `sigprocmask` 屏蔽 SIGCHLD，避免 handler 与前台 wait 抢同一个子进程

## 二、秦健超 — 命令解析与主控

> 负责文件：`main.c`、`parser.c`、`shell.c`、`input.c` 及对应头文件

### 2.1 负责的功能一览

| 功能 | 说明 | 关键函数 |
|------|------|----------|
| 主循环 | 启动初始化、循环读命令、退出清理 | `main()` |

### 2.2 main.c — 程序入口与主循环

`main.c` 是整个 Shell 的入口，做的事情很直白：**初始化 → 循环读命令 → 清理退出**。

#### 初始化阶段（启动时执行一次）

```c
// src/main.c:55-62

int main(void)
{
    executor_setup_signals();       // ① 张孙毓：注册 SIGCHLD 信号处理
    history_init(&g_history);      // ② 张孙毓：初始化历史记录环形缓冲区
    history_load(&g_history);      //    从 ~/.myshell_history 加载历史
    history_sync_readline(&g_history); // 同步到 readline（上下键可浏览）
    alias_init();                   // ③ 郭沛特：初始化别名链表
    completion_init();              // ④ 张孙毓：注册 Tab 补全回调
    // ...
```

这里调用了另外两位同学的初始化函数，体现了模块化协作。

#### 主循环（每条命令重复执行）

```c
// src/main.c:64-75

    for (;;) {
        char *raw = input_read_line();      // ① 秦健超：读入一行
        if (!raw)
            break;                          // Ctrl+D 或 EOF → 退出

        trim_inplace(raw);                  // 去掉首尾空格
        if (*raw && process_command(raw)) { // 非空行 → 处理
            free(raw);
            break;                          // process_command 返回 1 表示要退出
        }
        free(raw);
    }
```

`process_command()` 是核心处理函数，把读入的字符串变成可执行的命令：

```c
// src/main.c:30-53

static int process_command(const char *line)
{
    if (!line || !*line)
        return 0;

    // ① 郭沛特：别名展开（如 ll → ls -l），只替换行首第一个词
    char *expanded = alias_expand_line(line);
    if (!expanded) {
        fprintf(stderr, "myshell: out of memory\n");
        return 0;
    }

    // ② 秦健超：解析为 Pipeline 结构
    Pipeline pl;
    memset(&pl, 0, sizeof(pl));
    if (parse_line(expanded, &pl) < 0) {
        free(expanded);
        return 0;                       // 解析失败 → 跳过，不记历史
    }

    // ③ 张孙毓：记录原始输入到历史（注意是 raw 不是 expanded）
    history_add(&g_history, line);

    // ④ 秦健超：分发执行
    shell_execute(&pl);

    // ⑤ 秦健超：释放 Pipeline 内存
    parser_free_pipeline(&pl);
    free(expanded);
    return shell_should_exit();        // exit 命令会设置退出标志
}
```

**关键细节：** 历史记录保存的是用户**原始输入**（`line`），而不是别名展开后的内容（`expanded`）。这样 `history` 看到的是用户实际敲的命令，更符合直觉。

### 2.4 parser.c — 命令解析器

这是秦健超最核心的模块，负责把一行文本变成结构化的 `Pipeline`。

#### 解析流程总览

```
原始字符串: "grep hello a.txt | grep hello > out.txt &"
    │
    ① strip_background_flag()  识别行末 &，设置 background=1
    │  → "grep hello a.txt | grep hello > out.txt"
    │
    ② 按 | 分段（跳过引号内的 |）
    │  → 段1: "grep hello a.txt"
    │  → 段2: "grep hello > out.txt"
    │
    ③ 每段调用 parse_segment() 解析参数和重定向
    │  → cmds[0]: {argv: ["grep","hello","a.txt"]}
    │  → cmds[1]: {argv: ["grep","hello"], out_file: "out.txt"}
    │
    ④ 填入 Pipeline 结构
```

#### 入口函数 parse_line()

```c
// src/parser.c:171-244（核心逻辑）

int parse_line(const char *line, Pipeline *pl)
{
    if (!pl || !line)
        return -1;

    parser_free_pipeline(pl);           // 先清理上次的数据

    char *copy = strdup(line);          // 复制一份（解析过程会修改字符串）
    if (!copy)
        return -1;

    char *p = skip_space(copy);         // 跳过前导空格
    strip_background_flag(p, pl);       // ① 识别行末 &

    // ... 空行检查 ...

    char *seg = p;
    while (seg && *seg) {
        // ② 按 | 分段：遍历字符，遇到引号就跳过，遇到 | 就截断
        char *bar = NULL;
        int in_quote = 0;
        char quote = 0;
        for (char *q = seg; *q; q++) {
            if (in_quote) {
                if (*q == quote)
                    in_quote = 0;       // 引号结束
                continue;
            }
            if (*q == '"' || *q == '\'') {
                in_quote = 1;           // 进入引号
                quote = *q;
                continue;
            }
            if (*q == '|') {
                bar = q;                // 找到管道符
                break;
            }
        }

        if (bar)
            *bar = '\0';                // 在 | 处截断，得到一段

        // ③ 解析这一段
        if (parse_segment(seg, &pl->cmds[pl->count]) < 0) {
            // ... 错误处理 ...
        }
        pl->count++;

        seg = bar ? bar + 1 : NULL;     // ④ 移到下一段
    }

    free(copy);
    return pl->count > 0 ? 0 : -1;
}
```

**关键设计——引号内不拆分：** 注意上面 `for` 循环中对引号的处理。如果用户输入 `echo "a|b"`，引号内的 `|` 不会被当作管道符，而是作为参数的一部分。这避免了 `"a|b"` 被错误拆分成两段命令。

> **Linux 原理：** Shell 解析本质是**词法分析（tokenization）**——把字符流切成"词"（token）。空格、`|`、`>` 是分隔符，但引号内的字符具有"字面量"语义，分隔符失效。这与编译器词法分析器处理字符串字面量的思路相同。`read_next_arg()` 遇到 `'`/`"` 进入引号模式，引号内空格不会断词，因此 `alias dir='ls -l'` 整体是一个参数 token。

#### parse_segment() — 解析单段命令

`parse_segment()` 负责把一段字符串（如 `grep hello > out.txt`）解析为 `Command` 结构：

```c
// src/parser.c:101-150

static int parse_segment(char *seg, Command *cmd)
{
    memset(cmd, 0, sizeof(*cmd));
    cmd->argv[0] = NULL;

    char *p = seg;
    while (*p) {
        p = skip_space(p);              // 跳过空格
        if (!*p)
            break;

        if (*p == '<') {                // 输入重定向
            p++;
            if (read_redirect_path(&p, &cmd->in_file) < 0)
                return -1;
            continue;
        }
        if (*p == '>') {                // 输出重定向
            int append = 0;
            p++;
            if (*p == '>') {            // >> 追加模式
                append = 1;
                p++;
            }
            if (read_redirect_path(&p, &cmd->out_file) < 0)
                return -1;
            cmd->append = append;
            continue;
        }

        // 普通参数（支持引号）
        char *arg = NULL;
        if (read_next_arg(&p, &arg) < 0) {
            free(arg);
            return -1;
        }
        cmd->argv[cmd->argc++] = arg;   // 填入 argv 数组
        cmd->argv[cmd->argc] = NULL;    // 以 NULL 结尾
    }

    return cmd->argc > 0 ? 0 : -1;
}
```

#### read_next_arg() — 读取一个参数（支持引号）

```c
// src/parser.c:56-94

static int read_next_arg(char **pp, char **out)
{
    char *p = skip_space(*pp);
    if (!*p)
        return -1;

    char buf[MAX_TOKEN];
    size_t len = 0;

    while (*p) {
        if (*p == '<' || *p == '>' || *p == '|')
            break;                          // 遇到特殊符号，停止
        if (*p == '"' || *p == '\'') {      // 引号：整体作为一个参数
            char q = *p++;
            while (*p && *p != q)           // 把引号内的内容复制到 buf
                buf[len++] = *p++;
            if (*p != q)
                return -1;                  // 引号不匹配
            p++;
            continue;
        }
        if (isspace((unsigned char)*p))
            break;                          // 空格分隔
        buf[len++] = *p++;
    }

    buf[len] = '\0';
    *out = strdup(buf);                     // 复制为独立字符串
    *pp = p;
    return *out ? 0 : -1;
}
```

**举例：** 输入 `echo "hello world" foo`，解析出两个参数：
- `argv[1]` = `"hello world"`（引号内的空格保留）
- `argv[2]` = `"foo"`

### 3.2 I/O 重定向 — executor_setup_redirection()

重定向的原理很简单：**用 `dup2()` 把文件描述符"偷梁换柱"**。

```c
// src/executor.c:70-102

int executor_setup_redirection(const Command *cmd)
{
    // 输入重定向：cmd < file
    if (cmd->in_file) {
        int fd = open(cmd->in_file, O_RDONLY);  // 打开文件
        if (fd < 0) {
            perror(cmd->in_file);
            return -1;
        }
        dup2(fd, STDIN_FILENO);  // 把 fd 复制到 stdin(0)，之后读 stdin 就是读文件
        close(fd);               // 关掉原来的 fd（stdin 已经指向文件了）
    }

    // 输出重定向：cmd > file 或 cmd >> file
    if (cmd->out_file) {
        // append=0 用 O_TRUNC（覆盖），append=1 用 O_APPEND（追加）
        int flags = O_WRONLY | O_CREAT | (cmd->append ? O_APPEND : O_TRUNC);
        int fd = open(cmd->out_file, flags, 0644);  // 0644 = 用户读写、其他人只读
        if (fd < 0) {
            perror(cmd->out_file);
            return -1;
        }
        dup2(fd, STDOUT_FILENO);  // 把 fd 复制到 stdout(1)，之后写 stdout 就是写文件
        close(fd);
    }

    return 0;
}
```

**通俗解释 `dup2`：** Linux 每个进程有一张文件描述符表，0=stdin、1=stdout、2=stderr。`dup2(fd, 1)` 的意思是"把 fd 指向的文件，复制到编号 1 的位置"。之后所有写到 1（stdout）的内容，实际上写到了 fd 指向的文件。

```
重定向前：                        重定向后（cmd > file）：
stdout(1) ──→ 终端                stdout(1) ──→ file（dup2 偷梁换柱）
                                  fd(3)    ──→ file（已 close，但 stdout 仍指向 file）
```

### 3.3 单条内建命令执行 — executor_run_single_builtin()

这是最常走的路径，处理 `ls`、`cat` 等单条内建命令。根据是否有重定向/后台，分三种情况：

```c
// src/executor.c:270-311

int executor_run_single_builtin(Command *cmd, int background)
{
    if (!cmd->argv[0])
        return 1;

    // exit 命令：直接设置退出标志
    if (strcmp(cmd->argv[0], "exit") == 0)
        return run_builtin(cmd);

    // 情况1：前台 + 有重定向 → 不 fork，在父进程内做 dup2，做完再恢复
    if (!background && (cmd->in_file || cmd->out_file))
        return run_builtin_in_parent(cmd);

    // 情况2：前台 + 无重定向 → 最简单，直接执行
    if (!background && !cmd->in_file && !cmd->out_file)
        return run_builtin(cmd);

    // 情况3：后台 → fork 子进程执行
    pid_t pid = fork();
    if (pid < 0) {
        perror("fork");
        return 1;
    }
    if (pid == 0) {                         // 子进程
        reset_child_signals();              // 恢复默认信号处理
        if (executor_setup_redirection(cmd) < 0)
            _exit(1);
        int rc = run_builtin(cmd);
        fflush(stdout);
        fflush(stderr);
        _exit(rc);                          // 子进程退出（注意用 _exit 不用 exit）
    }

    // 父进程：后台模式
    if (background) {
        printf("[%d] running in background\n", (int)pid);
        fflush(stdout);
        return 0;                           // 不等待，直接返回
    }

    // 父进程：前台 fork 模式（等待子进程）
    int status = 0;
    if (wait_pid_retry(pid, &status) < 0) {
        perror("waitpid");
        return 1;
    }
    return WIFEXITED(status) ? WEXITSTATUS(status) : 1;
}
```

**三种情况对比：**

| 情况 | 条件 | 是否 fork | 原因 |
|------|------|-----------|------|
| 前台 + 重定向 | `echo hi > file` | 否 | 在父进程内 dup2，执行完恢复，避免 fork 开销 |
| 前台 + 无重定向 | `ls -l` | 否 | 最简单，直接执行 |
| 后台 | `ls -l &` | 是 | 需要 fork 出独立子进程，父进程不阻塞 |

#### run_builtin_in_parent() — 父进程内重定向

```c
// src/executor.c:104-132

static int run_builtin_in_parent(Command *cmd)
{
    int saved_in = -1;
    int saved_out = -1;
    int rc;

    // 保存原始 stdin/stdout，以便执行完后恢复
    if (cmd->in_file)
        saved_in = dup(STDIN_FILENO);    // 备份 stdin
    if (cmd->out_file)
        saved_out = dup(STDOUT_FILENO);  // 备份 stdout

    // 设置重定向（dup2）
    if ((cmd->in_file || cmd->out_file) && executor_setup_redirection(cmd) < 0) {
        rc = 1;
        goto restore;
    }

    rc = run_builtin(cmd);                // 执行内建命令（输出已重定向到文件）

restore:
    // 恢复原始 stdin/stdout
    if (saved_in >= 0) {
        dup2(saved_in, STDIN_FILENO);
        close(saved_in);
    }
    if (saved_out >= 0) {
        dup2(saved_out, STDOUT_FILENO);
        close(saved_out);
    }
    return rc;
}
```

**为什么不 fork？** 对于 `echo hi > file` 这种前台命令，fork 一个子进程只为了做重定向太浪费了。直接在父进程内 dup2 → 执行 → 恢复，效率更高。关键是执行完后要**恢复**原始的 stdin/stdout，否则后续命令的输出都会跑到文件里。

### 3.4 管道执行 — execute_pipeline() 和 run_pipeline_children()

管道是 Shell 最精妙的功能之一。以 `cmd1 | cmd2 | cmd3` 为例：

```
          pipe[0]              pipe[1]
cmd1 ──────────→ cmd2 ──────────→ cmd3
  stdout→pipe写端  pipe读端→stdin  stdout→pipe写端  pipe读端→stdin
```

每个 `pipe` 有两个端：写端（pipe[i][1]）和读端（pipe[i][0]）。前一段的 stdout 连到管道写端，后一段的 stdin 连到管道读端。

#### execute_pipeline() — 管道入口

```c
// src/executor.c:313-362

int execute_pipeline(Pipeline *pl)
{
    // ... 参数检查：每段必须是内建命令 ...

    if (pl->count == 1)
        return execute_single(&pl->cmds[0], pl->background);

    int n = pl->count;
    int pipes[MAX_PIPELINE][2];

    // ① 创建 n-1 条管道
    for (int i = 0; i < n - 1; i++) {
        if (pipe(pipes[i]) < 0) {          // 系统调用：创建管道
            perror("pipe");
            close_pipe_fds(pipes, i);
            return 1;
        }
    }

    // ② 后台管道：额外 fork 一层，整个管道在后台子进程中运行
    if (pl->background) {
        pid_t bg = fork();
        if (bg < 0) { /* ... 错误处理 ... */ }
        if (bg == 0) {                     // 后台子进程
            int rc = run_pipeline_children(pl, pipes);
            _exit(rc < 0 ? 1 : rc);
        }
        close_pipe_fds(pipes, n - 1);      // 父进程关闭所有管道 fd
        printf("[%d] running in background\n", (int)bg);
        return 0;                          // 不等待，立即返回
    }

    // ③ 前台管道：直接运行
    return run_pipeline_children(pl, pipes);
}
```

#### run_pipeline_children() — 创建子进程并连接管道

```c
// src/executor.c:208-235

static int run_pipeline_children(Pipeline *pl, int pipes[][2])
{
    int n = pl->count;
    pid_t pids[MAX_PIPELINE];
    int started = 0;

    for (int i = 0; i < n; i++) {
        pid_t pid = fork();                // 为每段命令 fork 一个子进程
        if (pid < 0) {
            perror("fork");
            kill_pids(pids, started);      // 出错时杀掉已启动的子进程
            return -1;
        }
        if (pid == 0) {                    // 子进程
            // 非首段：stdin ← 前一条管道的读端
            if (i > 0 && dup2(pipes[i - 1][0], STDIN_FILENO) < 0)
                _exit(1);
            // 非末段：stdout → 当前管道的写端
            if (i < n - 1 && dup2(pipes[i][1], STDOUT_FILENO) < 0)
                _exit(1);
            // 关闭所有管道 fd（dup2 已经复制了需要的端）
            close_pipe_fds(pipes, n - 1);
            // 执行命令（设置重定向 + 跑内建）
            run_command_child(&pl->cmds[i]);
        }
        pids[i] = pid;
        started++;
    }

    // 父进程关闭所有管道 fd（子进程已经持有需要的端了）
    close_pipe_fds(pipes, n - 1);
    return wait_pids(pids, n);             // 等待所有子进程
}
```

**以三段管道 `cmd1 | cmd2 | cmd3` 为例，每个子进程的连接：**

| 子进程 | stdin 连接 | stdout 连接 |
|--------|------------|------------|
| cmd1（i=0） | 终端（不变） | pipe[0] 写端 |
| cmd2（i=1） | pipe[0] 读端 | pipe[1] 写端 |
| cmd3（i=2） | pipe[1] 读端 | 终端（不变） |

**为什么父进程和子进程都要关闭管道 fd？** 管道写端全部关闭后，读端 `read()` 才会返回 0（EOF）。如果父进程不关闭写端，cmd2 的 `read()` 会一直阻塞，因为还有进程（父进程）持有写端。

### 3.5 子进程统一入口 — run_command_child()

```c
// src/executor.c:142-165

static void run_command_child(Command *cmd)
{
    if (!cmd || !cmd->argv[0]) {
        fprintf(stderr, "myshell: empty command\n");
        _exit(1);
    }

    reset_child_signals();                        // 恢复默认信号处理

    if (executor_setup_redirection(cmd) < 0)      // 设置重定向（> < >>）
        _exit(1);

    if (is_builtin(cmd->argv[0])) {               // 是内建命令
        if (strcmp(cmd->argv[0], "exit") == 0)
            _exit(0);                              // 子进程中的 exit 直接退出
        int rc = run_builtin(cmd);                 // 执行内建（郭沛特的代码）
        fflush(stdout);
        fflush(stderr);
        _exit(rc);                                 // 用 _exit 退出（不执行 atexit 等）
    }

    fprintf(stderr, "myshell: %s: command not found\n", cmd->argv[0]);
    _exit(127);                                    // 非内建 → command not found
}
```

**注意 `_exit` vs `exit`：** 子进程中用 `_exit()` 而非 `exit()`。`exit()` 会调用 `atexit` 注册的清理函数和刷新 stdio 缓冲区，可能和父进程冲突；`_exit()` 直接进入内核，更安全。

### 3.6 后台运行

后台运行的核心思想：**fork 后不等待，父进程立即返回提示符**。

- **单条命令**（`ls &`）：`executor_run_single_builtin()` 中 fork 后不 wait，打印 PID
- **管道命令**（`ls | grep txt &`）：`execute_pipeline()` 中额外 fork 一层，整个管道在后台子进程中运行

```c
// src/executor.c:256-259（单条后台命令）

    if (background) {
        printf("[%d] running in background\n", (int)pid);
        fflush(stdout);
        return 0;                           // 不调用 waitpid，立即返回
    }
```

后台子进程结束后，由 SIGCHLD 处理函数自动回收（见下节）。

### 3.7 子进程回收与 SIGCHLD 信号处理

这是本模块最精巧的部分。问题是：**后台子进程结束后会变成僵尸进程（zombie），需要有人回收**。

#### 方案设计

```c
// src/executor.c:14-21 — SIGCHLD 处理函数

static void sigchld_handler(int sig)
{
    (void)sig;
    int saved_errno = errno;                // 保存 errno（信号处理函数中不能用 perror）
    while (waitpid(-1, NULL, WNOHANG) > 0)  // 非阻塞回收所有已退出的子进程
        ;
    errno = saved_errno;                    // 恢复 errno
}
```

```c
// src/executor.c:23-31 — 注册信号处理

void executor_setup_signals(void)
{
    struct sigaction sa;
    memset(&sa, 0, sizeof(sa));
    sa.sa_handler = sigchld_handler;
    sigemptyset(&sa.sa_mask);
    sa.sa_flags = SA_RESTART;               // 被信号中断的系统调用自动重启
    sigaction(SIGCHLD, &sa, NULL);          // 注册 SIGCHLD 处理函数
}
```

#### 为什么不用 `signal(SIGCHLD, SIG_IGN)`？

`SIG_IGN` 会让内核自动回收子进程，看似方便，但有致命问题：前台 `waitpid(pid, ...)` 会返回 `ECHILD`（No child processes），因为内核已经把子进程回收了。这与实验要求"父进程 wait 前台命令"冲突。

#### 竞态问题与解决

前台 `waitpid` 和 SIGCHLD 处理函数可能竞争同一个子进程：

```
场景：前台 fork 了一个子进程
  ① 子进程退出 → 内核发送 SIGCHLD
  ② SIGCHLD 处理函数 waitpid(-1, WNOHANG) → 回收了子进程
  ③ 父进程 waitpid(pid, 0) → 返回 ECHILD（子进程已被处理函数回收）
```

解决方案：**在 waitpid 期间屏蔽 SIGCHLD**

```c
// src/executor.c:49-60 — 前台 wait 时屏蔽 SIGCHLD

static int wait_pid_retry(pid_t pid, int *status)
{
    sigset_t mask, oldmask;
    sigemptyset(&mask);
    sigaddset(&mask, SIGCHLD);
    sigprocmask(SIG_BLOCK, &mask, &oldmask);  // 屏蔽 SIGCHLD

    int rc = wait_pid_blocking(pid, status);  // 安全地 waitpid

    sigprocmask(SIG_SETMASK, &oldmask, NULL); // 恢复
    return rc;
}
```

屏蔽期间，SIGCHLD 信号被挂起（不会丢失），等 `waitpid` 完成并恢复掩码后才递送。这样前台 wait 不会被处理函数抢先。

### 3.8 history.c — 历史记录

#### 存储结构：环形缓冲区

```c
// include/history.h

#define HISTSIZE 1000

typedef struct {
    char *entries[HISTSIZE];  // 固定大小数组
    int   count;              // 当前条目数
    int   start;              // 环形缓冲区起始索引
} History;
```

用环形缓冲区的好处：满了之后自动覆盖最旧的条目，不需要移动数组。`start` 指向最旧条目的位置，新条目写入 `(start + count) % HISTSIZE`。

#### 核心函数

```c
// src/history.c:46-74 — 添加历史

void history_add(History *hist, const char *line)
{
    if (!line || !*line)
        return;                         // 空行不记录

    // 去重：和上一条相同则不记录
    if (hist->count > 0) {
        int last = (hist->start + hist->count - 1) % HISTSIZE;
        if (hist->entries[last] && strcmp(hist->entries[last], line) == 0)
            return;
    }

    // 缓冲区已满 → 覆盖最旧的
    if (hist->count == g_hist_limit) {
        char *dup = strdup(line);
        if (!dup)
            return;
        free(hist->entries[hist->start]);           // 释放最旧的
        hist->entries[hist->start] = dup;            // 写入新条目
        hist->start = (hist->start + 1) % HISTSIZE;  // start 前移
        add_history(line);                           // 同步到 readline
        return;
    }

    // 未满 → 直接追加
    int idx = (hist->start + hist->count) % HISTSIZE;
    hist->entries[idx] = strdup(line);
    if (hist->entries[idx]) {
        hist->count++;
        add_history(line);                           // 同步到 readline（上下键）
    }
}
```

#### cd 命令 — builtin_cd()

```c
// src/builtins.c:74-98

static int builtin_cd(Command *cmd)
{
    char *expanded = NULL;
    const char *dir;

    if (cmd->argc <= 1) {
        dir = getenv("HOME");             // 无参数 → 回家目录
    } else {
        expanded = expand_tilde(cmd->argv[1]);  // 展开 ~ 为 HOME
        dir = expanded ? expanded : cmd->argv[1];
    }

    if (!dir) {
        fprintf(stderr, "cd: HOME not set\n");
        free(expanded);
        return 1;
    }
    if (chdir(dir) != 0) {               // 系统调用：切换工作目录
        perror("cd");
        free(expanded);
        return 1;
    }
    free(expanded);
    return 0;
}
```

**为什么 cd 必须是内建命令？** `chdir()` 只改变**调用进程**的工作目录。如果 fork 子进程来执行 cd，子进程的目录变了，但父进程（Shell 本身）的目录没变。用户敲 `cd /tmp` 后再敲 `pwd`，会发现目录没变——这显然不对。所以 cd 必须在 Shell 进程内直接调用 `chdir()`。

### 4.3 builtin_ls.c — ls 命令（系统调用实现）

`ls` 的核心是三个系统调用：
- `opendir(path)`：打开目录
- `readdir(dir)`：逐个读取目录项
- `stat(path)`：获取文件信息（大小、权限、修改时间等）

```c
// src/builtin_ls.c:52-89 — 列出目录内容

static int list_dir(const char *path, int long_fmt, int show_all)
{
    DIR *d = opendir(path);              // 系统调用：打开目录
    if (!d) {
        // 如果不是目录，尝试当普通文件处理
        struct stat st;
        if (stat(path, &st) == 0 && !S_ISDIR(st.st_mode))
            return list_file(path, long_fmt);
        perror(path);
        return 1;
    }

    struct dirent *ent;
    int rc = 0;
    while ((ent = readdir(d))) {         // 系统调用：逐个读取目录项
        char full[4096];

        if (!show_all && ent->d_name[0] == '.')
            continue;                    // -a 未指定时，跳过隐藏文件

        if (long_fmt) {                  // -l 长格式
            // 拼接完整路径，用于 stat
            if (strcmp(path, ".") == 0)
                snprintf(full, sizeof(full), "%s", ent->d_name);
            else
                snprintf(full, sizeof(full), "%s/%s", path, ent->d_name);
            struct stat st;
            if (stat(full, &st) < 0) {   // 系统调用：获取文件信息
                perror(ent->d_name);
                rc = 1;
                continue;
            }
            ls_long(ent->d_name, &st);   // 输出长格式信息
        } else {
            printf("%s\n", ent->d_name); // 短格式：只输出文件名
        }
    }
    closedir(d);                         // 关闭目录
    return rc;
}
```

#### 长格式输出 — ls_long()

```c
// src/builtin_ls.c:11-36

static int ls_long(const char *name, struct stat *st)
{
    // 解析文件类型和权限，如 "drwxr-xr-x"
    char mode[11];
    mode[0] = S_ISDIR(st->st_mode) ? 'd' : '-';   // 目录 or 普通文件
    mode[1] = (st->st_mode & S_IRUSR) ? 'r' : '-'; // 用户读
    mode[2] = (st->st_mode & S_IWUSR) ? 'w' : '-'; // 用户写
    mode[3] = (st->st_mode & S_IXUSR) ? 'x' : '-'; // 用户执行
    // ... rwx for group and others ...

    // 格式化修改时间
    char timebuf[32];
    struct tm *tm_info = localtime(&st->st_mtime);
    if (tm_info)
        strftime(timebuf, sizeof(timebuf), "%b %e %H:%M", tm_info);
    else
        timebuf[0] = '\0';

    // 输出：权限 硬链接数 文件大小 时间 文件名
    printf("%s %3ld %8ld %s %s\n", mode, (long)st->st_nlink,
           (long)st->st_size, timebuf, name);
    return 0;
}
```

`struct stat` 是系统调用 `stat()` 填充的结构体，包含文件的几乎所有元数据。

### 4.4 builtin_cat.c — cat 命令（系统调用实现）

`cat` 的核心是三个系统调用：`open` → `read` → `write`。

```c
// src/builtin_cat.c:7-22 — 核心：把一个 fd 的内容复制到另一个 fd

static int copy_fd(int from, int to)
{
    char buf[4096];                      // 4KB 缓冲区
    ssize_t n;

    // 循环读取，直到 EOF（read 返回 0）
    while ((n = read(from, buf, sizeof(buf))) > 0) {
        ssize_t off = 0;
        // 可能一次 write 写不完，需要循环写
        while (off < n) {
            ssize_t w = write(to, buf + off, (size_t)(n - off));
            if (w < 0)
                return -1;               // 写出错
            off += w;
        }
    }
    return n < 0 ? -1 : 0;               // n < 0 表示 read 出错
}
```

```c
// src/builtin_cat.c:24-44 — 命令入口

int builtin_cat(Command *cmd)
{
    // 无参数 → 从 stdin 读（支持管道 cat | grep ...）
    if (cmd->argc <= 1)
        return copy_fd(STDIN_FILENO, STDOUT_FILENO) < 0 ? 1 : 0;

    int rc = 0;
    for (int i = 1; i < cmd->argc; i++) {
        int fd = open(cmd->argv[i], O_RDONLY);  // 系统调用：打开文件
        if (fd < 0) {
            perror(cmd->argv[i]);               // 文件不存在等错误
            rc = 1;
            continue;
        }
        if (copy_fd(fd, STDOUT_FILENO) < 0) {    // 读文件 → 写 stdout
            perror(cmd->argv[i]);
            rc = 1;
        }
        close(fd);                               // 系统调用：关闭文件
    }
    return rc;
}
```

**为什么 `write` 要循环？** `write` 不保证一次写完所有数据，可能只写了一部分（尤其是写管道或网络时）。所以要用循环确保全部写完。

### 4.5 builtin_grep.c — grep 命令

`grep` 逐行读取输入，匹配包含指定模式的行：

```c
// src/builtin_grep.c:24-50 — 逐行匹配

static int grep_stream(FILE *fp, const char *pattern, int ignore_case, int show_name,
                       const char *label)
{
    char *line = NULL;
    size_t cap = 0;
    ssize_t n;
    int matched = 0;

    while ((n = getline(&line, &cap, fp)) != -1) {  // 逐行读取
        if (n > 0 && line[n - 1] == '\n')
            line[n - 1] = '\0';                      // 去掉换行符

        // 匹配：-i 忽略大小写用 strcase_contains，否则用 strstr
        int hit = ignore_case ? strcase_contains(line, pattern)
                              : (strstr(line, pattern) != NULL);
        if (!hit)
            continue;

        if (show_name)                               // 多文件时显示文件名
            printf("%s:", label);
        fputs(line, stdout);
        putchar('\n');
        matched = 1;
    }

    free(line);
    return matched ? 0 : 1;                          // 匹配到返回 0，否则 1
}
```

```c
// src/builtin_grep.c:52-93 — 命令入口

int builtin_grep(Command *cmd)
{
    if (cmd->argc < 2) {
        fprintf(stderr, "grep: missing pattern\n");
        return 1;
    }

    const char *pattern = cmd->argv[1];
    int ignore_case = 0;
    int first_arg = 2;

    // 处理 -i 选项
    if (pattern[0] == '-') {
        for (const char *p = pattern + 1; *p; p++) {
            if (*p == 'i')
                ignore_case = 1;
        }
        if (cmd->argc < 3) {
            fprintf(stderr, "grep: missing pattern\n");
            return 1;
        }
        pattern = cmd->argv[2];    // -i 后面才是真正的 pattern
        first_arg = 3;
    }

    // 无文件参数 → 从 stdin 读（支持管道）
    if (first_arg >= cmd->argc)
        return grep_stream(stdin, pattern, ignore_case, 0, NULL);

    // 逐个文件匹配
    int rc = 1;
    int multi = cmd->argc - first_arg > 1;  // 多文件时显示文件名前缀
    for (int i = first_arg; i < cmd->argc; i++) {
        FILE *fp = fopen(cmd->argv[i], "r");
        if (!fp) {
            perror(cmd->argv[i]);
            rc = 1;
            continue;
        }
        if (grep_stream(fp, pattern, ignore_case, multi, cmd->argv[i]) == 0)
            rc = 0;                           // 只要有一个文件匹配到，返回 0
        fclose(fp);
    }
    return rc;
}
```

### 4.7 alias.c — 别名

#### 存储结构：链表

```c
// include/alias.h

typedef struct AliasNode {
    char *name;                 // 别名，如 "ll"
    char *value;                // 替换串，如 "ls -l"
    struct AliasNode *next;
} AliasNode;
```

#### 别名展开 — alias_expand_line()

```c
// src/alias.c:76-109

char *alias_expand_line(const char *line)
{
    if (!line || !*line)
        return NULL;

    // 提取行首第一个词（命令名）
    const char *p = line;
    while (*p && isspace((unsigned char)*p))
        p++;
    if (!*p)
        return strdup(line);

    char name[256];
    size_t i = 0;
    while (p[i] && !isspace((unsigned char)p[i]) && i + 1 < sizeof(name)) {
        name[i] = p[i];
        i++;
    }
    name[i] = '\0';

    // alias/unalias 命令本身不展开（否则 alias ll='ls -l' 会被误展开）
    if (strcmp(name, "alias") == 0 || strcmp(name, "unalias") == 0)
        return strdup(line);

    // 查找别名
    const AliasNode *al = alias_find(name);
    if (!al)
        return strdup(line);             // 没找到别名 → 原样返回

    // 拼接：别名值 + 剩余部分
    const char *rest = p + i;            // 命令名后面的部分（参数等）
    size_t new_len = strlen(al->value) + strlen(rest) + 1;
    char *out = malloc(new_len);
    if (!out)
        return NULL;
    snprintf(out, new_len, "%s%s", al->value, rest);  // 如 "ls -l" + " /tmp"
    return out;
}
```

**展开时机：** 在 `main.c` 的 `process_command()` 中、**解析之前**调用。展开后再交给 `parse_line()` 解析。只替换行首第一个词，避免误替换参数中的同名文本。

> **Linux 原理：** 别名展开发生在**解析前**（pre-parse），类似 bash 的 alias expansion 阶段。`alias`/`unalias` 命令本身不展开，否则 `alias ll='ls -l'` 会被误替换。展开只替换行首命令名，因为别名语义是"命令 shortcuts"，不是文本全局替换。

#### alias 命令 — builtin_alias()

`alias dir='ls -l'` 若被解析器拆成 `dir='ls` 和 `-l'` 两个 argv，`builtin_alias` 会**逐段合并**后再解析 `name=value`：

```c

}
```

## 五、系统调用 vs 系统命令对照

本项目严格遵守"系统调用可以，系统命令不行"的原则。下表展示每个命令的实现方式：

| 命令 | ❌ 套壳方式（禁止） | ✅ 本项目实现方式 | 核心系统调用 |
|------|-------------------|------------------|-------------|
| `ls` | `execvp("/bin/ls", ...)` | `opendir` + `readdir` + `stat` | `getdents`, `stat` |
| `cat` | `execvp("/bin/cat", ...)` | `open` + `read` + `write` | `open`, `read`, `write` |
| `grep` | `execvp("/bin/grep", ...)` | `fopen` + `getline` + `strstr` | `read`（via stdio） |
| `pwd` | `execvp("/bin/pwd", ...)` | `getcwd` | `getcwd` |
| `cd` | 无法用 exec 实现 | `chdir` | `chdir` |
| `echo` | `execvp("/bin/echo", ...)` | `putchar` + `fputs` + `getenv` | `write`（via stdio） |

**代码中搜索 `execvp`/`execl`/`system(``/`popen` 全部为 0 匹配**——本项目没有任何套壳调用。
